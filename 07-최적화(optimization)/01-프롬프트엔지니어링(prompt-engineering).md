# 효과적인 프롬프트 작성법

> Claude Code에서 원하는 결과를 빠르게 얻기 위한 프롬프트 엔지니어링 가이드입니다.

---

## 1. 프롬프트 작성 원칙

### 핵심 3원칙

1. **구체적으로** (Specific): 무엇을, 어디에, 어떻게 원하는지 명확하게
2. **맥락 제공** (Contextual): 프로젝트 상황, 제약 조건, 기존 코드 설명
3. **구조화** (Structured): 요구사항을 목록이나 단계로 나누어 전달

---

## 2. Before/After 프롬프트 예제

### 예제 1: 함수 작성

**나쁜 프롬프트:**
```
정렬 함수 만들어줘
```

**좋은 프롬프트:**
```
utils/sort.ts에 배열 정렬 유틸리티 함수를 만들어줘.
- 제네릭 타입 지원
- 정렬 기준 필드와 방향(asc/desc)을 파라미터로
- 안정 정렬(stable sort) 보장
- 기존 Array.prototype.sort 래퍼가 아닌 독립 함수
```

**포인트:** 파일 위치, 기술 요구사항, 제약 조건을 명시하면 원하는 결과를 한 번에 얻을 수 있습니다.

### 예제 2: 버그 수정

**나쁜 프롬프트:**
```
로그인이 안 돼
```

**좋은 프롬프트:**
```
로그인 시 "401 Unauthorized" 에러가 발생해.
- POST /api/auth/login 엔드포인트
- 요청 body: { email: "test@test.com", password: "1234" }
- 서버 로그: "JWT_SECRET is undefined"
- 환경: 로컬 개발 서버 (npm run dev)
src/middleware/auth.ts 파일을 확인하고 원인을 찾아 수정해줘.
```

**포인트:** 에러 메시지, 재현 단계, 관련 파일을 제공하면 즉시 원인을 파악할 수 있습니다.

### 예제 5: API 설계

**나쁜 프롬프트:**
```

```

```

- 평점(1-5), 내용, 이미지(최대 3장) 필드
- 기존 Express + Prisma 패턴에 맞춰서
```

## 3. 고급 프롬프트 패턴

> 아래 패턴들은 [프롬프트 구조화 6대 원칙](#프롬프트-구조화-6대-원칙)의 개별 섹션을 실전에서 활용하는 예시입니다.

### 단계별 지시

복잡한 작업은 단계를 나누어 지시합니다:

```
사용자 인증 시스템을 구현해줘. 다음 순서로 진행해:

1단계: Prisma 스키마에 User, Session 모델 추가
2단계: 회원가입 API (이메일 중복 검사, 비밀번호 해싱)
3단계: 로그인 API (JWT 발급)
4단계: 인증 미들웨어 (토큰 검증)
5단계: 각 단계별 테스트 작성

각 단계 완료 후 결과를 보여주고 다음으로 진행해.
```

## 4. 피해야 할 패턴

| 피해야 할 것 | 이유 | 대안 |
|-------------|------|------|
| 너무 긴 프롬프트 (500자+) | 핵심이 흐려짐 | 요구사항을 목록으로 정리 |
| 모호한 표현 ("좋게", "깔끔하게") | 주관적 해석 | 구체적 기준 제시 |
| 한 번에 너무 많은 요청 | 품질 저하 | 단계별로 나누어 요청 |
| 파일 경로 생략 | 잘못된 위치에 생성 | 항상 경로 명시 |
| 기존 코드 패턴 무시 | 일관성 깨짐 | "기존 패턴을 따라서" 추가 |

## 프롬프트 구조화 6대 원칙

좋은 프롬프트는 아래 6가지 요소를 빠짐없이 갖춘다. 하나라도 빠지면 에이전트가 추측하거나 범위를 스스로 확장한다.

| # | 원칙 | 핵심 |
|---|------|------|
| 1 | **역할 (Role)** | 에이전트의 기본 역할을 고정 — "시니어 개발자", "보안 엔지니어" 등 |
| 2 | **참조 (Context)** | 먼저 읽을 문서/파일을 경로로 명시 — 추측과 환각 방지 |
| 3 | **목적 (Intent)** | "왜 하는지" 한 줄 — 트레이드오프 판단 기준 제공 |
| 4 | **범위 (Scope)** | 건드려도 되는 것 / 안 되는 것 — 스코프 확장 방지 |
| 5 | **완료 기준 (Done Criteria)** | 체크박스 3~7개 — 무한 루프와 거짓 완료 방지 |
| 6 | **제약 (Constraints)** | 성능, 보안, 호환성, 금지 패턴 — 경계 조건 고정 |

### 공통 안티패턴

| 하지 마라 | 이유 |
|----------|------|
| "이거 좀 봐줘" | 범위가 무한. 에이전트는 소설을 씀. |
| "최대한 예쁘게" | 판단 기준이 없음. |
| "이런 거 알지?" | 에이전트 메모리는 세션마다 리셋. |
| 파일 경로 없이 "그 컴포넌트" | 추측 → 엉뚱한 파일 수정. |
| 완료 기준 없음 | 언제 끝났는지 판단 불가. |

## 범용 프롬프트 템플릿 (7섹션)

위 6대 원칙에 **출력 형식(Output Format)** 을 추가한 실전 템플릿이다. 복사해서 빈 칸을 채운 뒤 에이전트에 그대로 붙여넣는다.

```markdown
## 1. 역할 (Role)
너는 이 프로젝트의 시니어 개발자다.
`CLAUDE.md`와 `docs/` 하위 문서가 단일 진실 공급원이다.
문서에 없는 사실을 추정해야 할 경우, 먼저 질문하라.

## 2. 참조 (Context)
먼저 다음 파일을 읽고 시작하라.
- CLAUDE.md
- docs/PRD.md  §<해당 섹션>
- docs/architecture.md  §<해당 섹션>
- docs/erd.md  (스키마 관련일 때만)
- <관련 소스 파일 경로들>

## 3. 목적 (Intent)
<한 문장: 이 작업을 왜 하는가 — 유저 가치 또는 기술 이유>

## 4. 범위 (Scope)
### 포함
- <건드려도 되는 파일/모듈>
- <건드려도 되는 동작>

### 제외 (Out of Scope)
- <건드리지 말아야 할 파일/모듈>
- <이번 PR에서 하지 말 것>

## 5. 완료 기준 (Done Criteria)
- [ ] <검증 가능한 조건 1>
- [ ] <검증 가능한 조건 2>
- [ ] <검증 가능한 조건 3>
- [ ] 관련 문서(PRD/erd/architecture) 동기화
- [ ] 기존 테스트 모두 통과
- [ ] 새 로직에 대한 테스트 추가

## 6. 제약 (Constraints)
- 금지: <예: any 타입, console.log, 신규 의존성>
- 유지: <예: 공개 API 시그니처, 에러 코드>
- 성능: <예: p95 < 200ms>
- 보안: <예: 사용자 입력 이스케이프>

## 7. 출력 형식 (Output Format)
먼저 다음 순서로 답하라.
1. 이해한 내용 요약 (2~3줄)
2. 변경할 파일 목록 + 각 파일의 변경 의도
3. 내가 승인하면 → 코드 작성
4. 마지막에 완료 기준 체크리스트 체크 결과
```

### 왜 7개 섹션인가

| 섹션 | 없으면 생기는 문제 |
|------|------------------|
| 1. 역할 | 에이전트가 어떤 톤/수준으로 답할지 흔들림 |
| 2. 참조 | 문서를 안 읽고 추측 → 환각 |
| 3. 목적 | 세부 판단 실수 (트레이드오프 잘못 선택) |
| 4. 범위 | 스코프 확장 |
| 5. 완료 기준 | 무한 루프 / "다 됐어요" 거짓말 |
| 6. 제약 | 금지 패턴 침범 |
| 7. 출력 형식 | 코드만 훅 쏟아내서 리뷰 불가 |

### 축약 버전 (작은 작업용)

1~2파일 수준의 작은 작업에선 섹션을 압축해도 된다. 대신 **삭제하지 말고 한 줄로 줄인다**.

```markdown
## 목적
<한 줄>

## 참조
- <파일 1>
- <파일 2>

## 완료 기준
- [ ] <조건 1>
- [ ] <조건 2>

## 제약
<한 줄>
```

> 에이전트가 바로 코드를 쏟아내면, 잘못된 파일에 잘못된 변경을 쓴 뒤에야 알게 된다.
> **"변경 계획"을 먼저 말하게 하면** 코드 생성 전에 잘못을 잡을 수 있다 — 이 한 단계로 실패율이 체감상 절반으로 준다.

> 작업 유형별 전용 템플릿은 [04-워크플로우](../04-워크플로우(workflows)/) 폴더의 각 흐름 문서를 참조하세요.
