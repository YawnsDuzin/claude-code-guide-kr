# 디버깅 워크플로우

Claude Code를 활용하여 체계적으로 버그를 찾고 해결하는 방법을 소개합니다. 에러 메시지 분석부터 성능 문제, 메모리 누수, 경쟁 조건까지 다양한 디버깅 시나리오를 다룹니다.

---

## 1. 에러 메시지 분석

에러 메시지와 스택 트레이스를 기반으로 문제를 진단합니다.

### 기본 접근법

```
다음 에러가 발생해. 원인을 분석하고 수정해줘:

Error: ECONNREFUSED 127.0.0.1:5432
    at TCPConnectWrap.afterConnect [as oncomplete] (net.js:1141:16)
    at Protocol._enqueue (node_modules/pg-protocol/...)
    at Client.query (node_modules/pg/lib/client.js:...)
    at UserRepository.findAll (src/repositories/userRepository.ts:45:28)
```

### 상세 분석 요청

```
이 에러를 다음 관점에서 분석해줘:
1. 에러의 직접적인 원인
2. 이 에러가 발생할 수 있는 모든 시나리오
3. 관련 코드의 에러 처리가 적절한지
4. 재발 방지를 위한 개선 방안
```

### Claude의 예상 동작

1. 에러 메시지와 에러 코드를 해석
2. 스택 트레이스에서 프로젝트 코드 위치를 식별
3. 해당 파일을 읽어 문맥 파악
4. 관련 설정 파일(DB 연결 설정 등)을 확인
5. 근본 원인과 수정 방법을 제시

### 실전 예시

```
# 타입 에러 분석
"TypeScript 컴파일 에러가 여러 개 발생해. tsc 출력을 붙여넣을게:
[에러 출력 붙여넣기]
각 에러의 원인과 수정 방법을 알려줘"

# 런타임 에러 분석
"프로덕션에서 다음 에러가 발생해. Sentry 로그를 붙여넣을게:
[Sentry 로그]
어떤 사용자 행동이 이 에러를 유발하는지 추론해줘"
```

## 2. 로그 기반 디버깅

로그 데이터를 분석하여 문제를 추적합니다.

### 프롬프트 예시

```
다음 로그에서 문제를 찾아줘:

[2024-01-15 10:23:45] INFO  Request received: POST /api/orders
[2024-01-15 10:23:45] DEBUG Validating order payload
[2024-01-15 10:23:46] DEBUG Checking inventory for product: SKU-001
[2024-01-15 10:23:46] INFO  Inventory check passed
[2024-01-15 10:23:48] DEBUG Processing payment...
[2024-01-15 10:23:53] WARN  Payment gateway timeout (5000ms exceeded)
[2024-01-15 10:23:53] ERROR Payment failed: TimeoutError
[2024-01-15 10:23:53] DEBUG Rolling back inventory reservation
[2024-01-15 10:23:53] ERROR Rollback failed: Connection pool exhausted

문제점을 분석하고 해결 방안을 제안해줘.
```

### 전략적 로그 추가 요청

```
주문 생성 과정에서 간헐적으로 실패가 발생해.
문제를 추적하기 위해 src/services/orderService.ts에 전략적인 로그를 추가해줘.

다음 정보를 포함해줘:
- 각 단계의 시작/완료 시간
- 외부 서비스 호출의 요청/응답
- 에러 발생 시 컨텍스트 정보 (주문 ID, 사용자 ID 등)
```

### Claude의 예상 동작

- 로그의 시간순 흐름을 분석
- 비정상적인 지연, 에러 패턴을 식별
- 로그에 누락된 정보를 파악
- 추가 로그가 필요한 위치를 제안
- 기존 로그 레벨이 적절한지 검토

## 3. 성능 병목 식별

응답 시간이 느리거나 리소스 사용량이 높은 원인을 찾습니다.

### 프롬프트 예시

```
이 API 엔드포인트의 응답 시간이 3초 이상이야. 병목을 찾아줘:

엔드포인트: GET /api/dashboard
증상: 평상시 200ms, 피크 시간대에 3-5초
관련 코드: src/controllers/dashboardController.ts

코드를 분석해서:
1. 병목이 될 수 있는 부분을 식별
2. 각 부분의 예상 소요 시간을 추정
3. 최적화 방안을 우선순위별로 제안
```

### 데이터베이스 성능 분석

```
다음 쿼리가 느려. 실행 계획을 분석하고 최적화해줘:

SELECT u.*, COUNT(o.id) as order_count, SUM(o.total) as total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.created_at > '2024-01-01'
GROUP BY u.id
ORDER BY total_spent DESC
LIMIT 100;

현재 실행 시간: 12초
테이블 크기: users 100만 행, orders 500만 행
```

### 프론트엔드 성능 분석

```
이 페이지의 초기 로딩이 느려. 다음 관점에서 분석해줘:
1. 불필요한 리렌더링
2. 큰 번들 사이즈의 원인
3. 최적화되지 않은 이미지나 리소스
4. 지연 로딩(lazy loading)이 필요한 컴포넌트
5. API 호출 패턴 (워터폴 문제 등)
```

### Claude의 예상 동작

- 코드의 시간 복잡도를 분석
- 데이터베이스 쿼리 패턴을 점검 (N+1 문제 등)
- 네트워크 호출의 순차/병렬 처리 여부 확인
- 캐싱 기회를 식별
- 구체적인 최적화 코드를 제안

## 4. 메모리 누수 탐지

메모리 사용량이 지속적으로 증가하는 문제를 찾습니다.

### 프롬프트 예시

```
이 Node.js 서버에서 메모리 누수가 의심돼.
서비스 시작 후 시간이 지나면 메모리 사용량이 계속 증가해.

다음 부분을 중점적으로 검토해줘:
1. 이벤트 리스너가 제거되지 않는 곳
2. setInterval/setTimeout이 정리되지 않는 곳
3. 클로저에서 큰 객체를 참조하고 있는 곳
4. 캐시가 무한정 커지는 곳
5. 스트림이 적절히 닫히지 않는 곳
6. 전역 변수에 데이터가 계속 쌓이는 곳
```

### React 메모리 누수

```
이 React 컴포넌트에서 메모리 누수 경고가 발생해:
"Can't perform a React state update on an unmounted component"

관련 컴포넌트들을 분석해서 다음을 확인해줘:
1. 언마운트 후에도 실행되는 비동기 작업
2. cleanup이 없는 useEffect
3. 구독(subscription)이 해제되지 않는 곳
```

### Claude의 예상 동작

- 이벤트 리스너 등록/해제 패턴을 확인
- 타이머와 인터벌의 정리 여부를 검증
- 클로저의 참조 체인을 분석
- 캐시 전략의 만료 정책을 확인
- 수정 코드와 메모리 프로파일링 방법을 제안

## 5. 네트워크 이슈 디버깅

API 통신, WebSocket, 외부 서비스 연동 문제를 해결합니다.

### 프롬프트 예시

```
프론트엔드에서 API 호출이 간헐적으로 실패해.
다음 정보를 바탕으로 원인을 분석해줘:

증상:
- 10번 중 1-2번 정도 실패
- 에러: "Network Error" (CORS 에러는 아님)
- 특정 시간대에 더 많이 발생
- 새로고침하면 정상 동작

관련 코드:
- API 클라이언트: src/lib/apiClient.ts
- 인터셉터: src/lib/interceptors.ts
- 환경 설정: src/config/api.ts

다음을 확인해줘:
1. 타임아웃 설정이 적절한지
2. 재시도(retry) 로직이 있는지
3. 연결 풀 설정이 적절한지
4. 에러 처리가 모든 케이스를 커버하는지
```

### WebSocket 디버깅

```
WebSocket 연결이 자주 끊어져. 관련 코드를 분석해줘:
1. 연결 유지(heartbeat/ping-pong) 구현이 있는지
2. 재연결 로직이 적절한지
3. 연결 상태 관리가 올바른지
4. 메시지 큐잉이 구현되어 있는지 (연결 끊김 시)
```

### Claude의 예상 동작

- 네트워크 관련 설정 파일을 확인
- HTTP 클라이언트의 설정(타임아웃, 재시도 등)을 검토
- 에러 처리 로직의 완전성을 평가
- CORS, 인증, 프록시 설정을 확인
- 개선된 에러 처리와 재시도 로직을 제안

## 6. 경쟁 조건(Race Condition) 식별

동시성 문제로 인한 간헐적 버그를 찾습니다.

### 프롬프트 예시

```
다음 상황에서 간헐적으로 데이터 불일치가 발생해:

상황: 두 사용자가 동시에 같은 상품의 재고를 차감할 때
증상: 재고가 1개인데 두 주문이 모두 성공하는 경우가 있음

관련 코드: src/services/inventoryService.ts

다음을 분석해줘:
1. 경쟁 조건이 발생할 수 있는 코드 경로
2. 트랜잭션 격리 수준이 적절한지
3. 낙관적/비관적 락이 필요한지
4. 수정 방안과 구현 코드
```

### 프론트엔드 경쟁 조건

```
검색 기능에서 이전 검색 결과가 나중에 표시되는 문제가 있어.
빠르게 검색어를 입력하면 "apple"로 검색했는데 "app"의 결과가 표시돼.

관련 코드를 분석하고 다음 해결 방안을 구현해줘:
1. 요청 취소 (AbortController)
2. 응답 순서 보장
3. 디바운싱 적용
```

### Claude의 예상 동작

- 공유 자원에 대한 동시 접근 패턴을 분석
- 트랜잭션 범위와 격리 수준을 확인
- 비동기 작업의 실행 순서를 추적
- 적절한 동기화 메커니즘을 제안
- 경쟁 조건을 재현하는 테스트 코드도 함께 제공

## 7. 단계별 디버깅 방법론

Claude와 함께하는 체계적인 디버깅 프로세스입니다.

### Step 1: 문제 정의

```
다음 버그를 보고합니다:
- 현상: [구체적인 증상]
- 예상 동작: [올바른 동작]
- 재현 절차: [재현 방법]
- 발생 빈도: [항상 / 간헐적]
- 최근 변경사항: [관련될 수 있는 변경]
```

### Step 2: 가설 수립

```
이 버그의 원인이 될 수 있는 가설을 세워줘.
가능성이 높은 순서로 정리하고, 각 가설을 검증하는 방법도 알려줘.
```

### Step 3: 가설 검증

```
첫 번째 가설을 검증해보자.
관련 코드를 확인하고 이 가설이 맞는지 분석해줘.
필요하면 확인용 로그를 추가해줘.
```

### Step 4: 수정 및 검증

```
원인을 찾았으니 수정해줘.
수정 후 다음을 확인해줘:
1. 원래 버그가 해결되는지
2. 사이드 이펙트가 없는지
3. 관련 테스트가 통과하는지
4. 동일한 패턴의 버그가 다른 곳에도 있는지
```

## 8. 전략적 로그 추가

Claude를 활용하여 효과적인 디버깅 로그를 추가합니다.

### 프롬프트 예시

```
주문 처리 흐름에 디버깅을 위한 console.log를 전략적으로 추가해줘.

조건:
- 각 주요 단계의 시작/완료 시점
- 함수 입력/출력값
- 조건 분기 시 어떤 분기를 탔는지
- 에러 발생 시 전체 컨텍스트
- 나중에 쉽게 제거할 수 있도록 [DEBUG] 접두사 사용

파일: src/services/orderService.ts
```

### 조건부 디버깅 로그

```
프로덕션에서도 안전하게 사용할 수 있는 디버그 로깅을 추가해줘:
- 환경 변수로 on/off 제어
- 구조화된 로그 형태 (JSON)
- 요청 ID를 포함하여 추적 가능하도록
- 민감 정보(비밀번호, 토큰)는 마스킹
```

### Python 디버깅 예시

```
이 Django 뷰에서 간헐적으로 500 에러가 발생해.
디버깅을 위해 다음 위치에 로그를 추가해줘:

1. 요청 파라미터 로깅
2. DB 쿼리 결과 로깅
3. 외부 API 호출 전후
4. 예외 발생 지점의 로컬 변수 상태

Python logging 모듈을 사용하고, 기존 로거 설정을 따라줘.
```

### Claude의 예상 동작

- 코드의 실행 흐름을 분석하여 핵심 지점을 식별
- 기존 로깅 프레임워크가 있으면 그것을 활용
- 적절한 로그 레벨(debug, info, warn, error)을 지정
- 로그 메시지에 충분한 컨텍스트를 포함
- 성능에 미치는 영향을 최소화

## 디버깅 체크리스트

빠르게 참고할 수 있는 디버깅 접근법 체크리스트입니다:

### 에러 발생 시

1. 에러 메시지와 스택 트레이스 전체를 Claude에게 전달
2. 발생 조건과 재현 방법을 함께 제공
3. 최근 변경사항이 있다면 언급

### 성능 문제 시

1. 정상 시와 문제 시의 수치를 비교하여 제공
2. 특정 조건(데이터 양, 동시 사용자 수)을 명시
3. 프로파일링 결과가 있다면 함께 제공

### 간헐적 버그 시

1. 발생 빈도와 패턴을 설명
2. 동시성/비동기 관련 가능성을 언급
3. 전략적 로그 추가를 먼저 요청

## 핵심 정리

| 디버깅 유형 | 핵심 접근법 | 제공해야 할 정보 |
|------------|-----------|----------------|
| 에러 분석 | 스택 트레이스 분석 | 에러 메시지, 재현 절차 |
| 로그 기반 | 로그 패턴 분석 | 관련 로그, 시간 정보 |
| 성능 병목 | 코드 흐름 분석 | 응답 시간, 발생 조건 |
| 메모리 누수 | 리소스 관리 검토 | 증상, 메모리 사용 추이 |
| 네트워크 문제 | 통신 설정 검토 | 에러 유형, 발생 빈도 |
| 경쟁 조건 | 동시성 패턴 분석 | 재현 시나리오, 데이터 상태 |

> **팁**: 디버깅 시 가장 중요한 것은 충분한 컨텍스트를 제공하는 것입니다. "안 돼요"보다는 "어떤 조건에서 어떤 에러가 발생하며, 예상 동작은 이것인데 실제로는 이렇게 동작한다"가 훨씬 효과적입니다.
