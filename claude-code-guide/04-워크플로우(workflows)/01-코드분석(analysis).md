# 코드 분석 워크플로우

Claude Code를 활용한 체계적인 코드 분석 방법을 소개합니다. 레거시 코드 이해부터 보안 감사까지, 실무에서 바로 활용할 수 있는 분석 워크플로우를 다룹니다.

---

## 1. 레거시 코드 이해

오래된 프로젝트나 처음 접하는 코드베이스를 빠르게 파악해야 할 때 사용합니다.

### 기본 프롬프트

```
이 프로젝트의 전체 구조를 설명해줘
```

### 상세 프롬프트 예시

```
이 프로젝트의 디렉토리 구조, 주요 모듈, 데이터 흐름을 분석해줘.
특히 다음 사항을 포함해줘:
1. 진입점(entry point)이 어디인지
2. 핵심 비즈니스 로직이 어디에 있는지
3. 외부 서비스와의 연동 포인트
4. 설정 파일들의 역할
```

```
src/services/ 디렉토리의 코드를 분석해줘. 각 서비스가 어떤 책임을 가지고 있고,
서로 어떻게 의존하는지 정리해줘.
```

```
이 프로젝트에서 사용하는 디자인 패턴들을 찾아서 설명해줘.
각 패턴이 어디에 적용되어 있고, 왜 그 패턴을 선택했는지 추론해줘.
```

### Claude의 예상 동작

- 프로젝트 루트의 `package.json`, `pyproject.toml`, `Cargo.toml` 등 설정 파일을 먼저 확인
- 디렉토리 구조를 탐색하여 모듈 간 관계를 파악
- 주요 파일의 import/export 관계를 추적
- 계층별(컨트롤러, 서비스, 모델 등) 역할을 분류하여 제시
- 전체 아키텍처를 요약하여 설명

### 실전 활용 팁

```
# 특정 기능의 코드 흐름을 추적할 때
"사용자 로그인 요청이 들어왔을 때 코드가 어떤 순서로 실행되는지 추적해줘"

# 특정 파일의 역할을 이해할 때
"src/middleware/auth.ts 파일이 전체 시스템에서 어떤 역할을 하는지 설명해줘"

# 설정 파일을 이해할 때
"webpack.config.js의 각 설정이 무엇을 의미하는지 설명해줘"
```

> **팁**: 프로젝트가 클 경우, 특정 디렉토리나 모듈을 지정하면 더 정확한 분석 결과를 얻을 수 있습니다. 예: `src/auth/ 모듈의 인증 흐름을 분석해줘`

프로젝트의 설계 패턴과 구조적 특성을 분석합니다.

### 기본 프롬프트

```
이 앱의 아키텍처 패턴을 분석해줘
```

### 상세 프롬프트 예시

```
이 프로젝트의 아키텍처를 다음 관점에서 분석해줘:
1. 사용된 디자인 패턴 (MVC, MVVM, Clean Architecture 등)
2. 계층(layer) 구조와 각 계층의 책임
3. 의존성 방향이 올바른지
4. SOLID 원칙 준수 여부
5. 개선이 필요한 아키텍처 문제점
```

```
이 프로젝트가 클린 아키텍처를 따르고 있는지 분석해줘.
각 레이어(도메인, 유스케이스, 인프라)가 제대로 분리되어 있는지,
의존성 방향이 올바른지 확인해줘.
```

```
이 마이크로서비스 간의 통신 패턴을 분석해줘.
동기/비동기 호출이 어디에 사용되고, 장애 전파 가능성이 있는 지점을 찾아줘.
```

### Claude의 예상 동작

- 디렉토리 구조와 명명 규칙을 분석하여 아키텍처 패턴 식별
- 모듈 간 의존성 방향을 확인
- API 라우트 정의와 미들웨어 구조 파악
- 계층 간 책임 분리가 적절한지 평가
- 구체적인 개선 제안을 코드 예시와 함께 제공

### 실전 활용 팁

```
# 프론트엔드 상태 관리 분석
"이 React 앱의 상태 관리 패턴을 분석하고 개선점을 제안해줘"

# API 설계 분석
"REST API 엔드포인트 설계가 일관성 있는지 분석해줘"

# 분석 후 개선 요청
"분석 결과를 바탕으로 아키텍처 개선 방안을 우선순위와 함께 제안해줘"
```

## 3. 버그 원인 분석 (Root Cause Analysis)

에러 메시지나 비정상 동작의 근본 원인을 추적합니다.

### 기본 프롬프트

```
이 에러의 원인을 추적해줘
```

### 상세 프롬프트 예시

```
다음 에러가 프로덕션에서 발생하고 있어. 원인을 추적해줘:

TypeError: Cannot read properties of undefined (reading 'map')
    at UserList (src/components/UserList.tsx:24:18)
    at renderWithHooks (node_modules/react-dom/...)

발생 조건:
- 사용자 목록 페이지에서 새로고침할 때
- API 응답이 늦을 때 간헐적으로 발생

이 컴포넌트에 전달되는 데이터 흐름을 따라가서 어디서 undefined가 발생하는지 찾아줘.
```

```
API 호출 시 간헐적으로 504 Gateway Timeout이 발생해.
관련 코드를 분석해서 타임아웃이 발생할 수 있는 모든 지점을 찾아줘.
특히 데이터베이스 쿼리와 외부 API 호출 부분을 중점적으로 봐줘.
```

```
다음 테스트가 CI에서만 실패하고 로컬에서는 통과해:

FAIL src/utils/date.test.ts
  Expected: "2024-03-15"
  Received: "2024-03-14"

타임존 관련 이슈인지 분석하고, 수정 방법을 제안해줘.
```

### Claude의 예상 동작

1. 에러 메시지와 스택 트레이스를 분석
2. 해당 파일(`UserList.tsx`)의 24번째 줄을 확인
3. 관련 데이터 흐름을 추적 (API 호출 → 상태 업데이트 → 렌더링)
4. 발생 조건을 고려하여 근본 원인을 식별
5. 가능한 원인을 우선순위로 나열하고 수정 방법을 구체적으로 제안

### 실전 활용 팁

```
# 에러 로그와 함께 분석 요청
"프로덕션에서 다음 에러 로그가 반복되고 있어. 원인과 해결책을 찾아줘:
[에러 로그 붙여넣기]"

# 재현 조건이 불분명할 때
"이 에러가 간헐적으로 발생하는데, 어떤 조건에서 발생할 수 있는지 코드를 분석해서 가능한 시나리오를 알려줘"

# 회귀(regression) 버그 분석
"최근 커밋 이후 이 기능이 깨졌어. 관련 변경사항을 분석해줘"
```

> **팁**: 에러 메시지와 함께 재현 조건(특정 입력값, 환경 등)을 알려주면 분석 정확도가 크게 올라갑니다. 스택 트레이스 전체를 붙여넣는 것이 부분만 넣는 것보다 효과적입니다.

## 4. 의존성 분석

프로젝트의 의존성을 점검하고 불필요한 패키지를 식별합니다.

### 기본 프롬프트

```
사용하지 않는 의존성을 찾아줘
```

### 상세 프롬프트 예시

```
이 프로젝트의 의존성을 분석해줘:
1. package.json에 있지만 실제로 import되지 않는 패키지
2. devDependencies에 있어야 하는데 dependencies에 있는 패키지
3. 중복 기능을 하는 패키지 (예: moment와 dayjs 동시 사용)
4. 보안 취약점이 알려진 패키지
5. 메이저 버전 업데이트가 필요한 패키지
```

```
번들 사이즈에 가장 큰 영향을 미치는 의존성 TOP 5를 찾고,
더 가벼운 대안이 있으면 제안해줘.
```

```
이 프로젝트의 의존성 트리를 분석해서 보안 취약점이 있는 패키지가 있는지 확인해줘.
대안 패키지도 함께 제안해줘.
```

### Claude의 예상 동작

- `package.json` (또는 `requirements.txt`, `go.mod` 등)을 읽기
- 프로젝트 전체에서 각 패키지의 import/require 문을 검색
- 사용되지 않는 의존성 목록을 작성
- 중복 기능을 가진 패키지를 식별
- 잘못 분류된 의존성을 식별
- 제거 또는 수정 방법을 안내

### 실전 활용 팁

```
# 의존성 충돌 해결
"npm install 시 peer dependency 충돌이 발생해. 해결 방법을 찾아줘"

# 버전 호환성 확인
"Node.js 20으로 업그레이드할 때 호환성 문제가 있는 패키지를 찾아줘"

# 제거 전 영향 확인
"해당 패키지를 제거하면 영향받는 코드를 모두 찾아줘"
```

> **팁**: 의존성 제거 전에 반드시 영향 범위를 확인하세요. 간접 의존성(다른 패키지의 peer dependency)으로 사용되는 경우도 있습니다.

## 5. 보안 감사

코드에 존재할 수 있는 보안 취약점을 체계적으로 검토합니다.

### 기본 프롬프트

```
보안 취약점을 검토해줘
```

### 상세 프롬프트 예시

```
이 프로젝트의 보안을 다음 항목 기준으로 감사해줘:

1. SQL Injection 가능성
2. XSS (Cross-Site Scripting) 취약점
3. 인증/인가 로직의 허점
4. 민감 정보 하드코딩 (API 키, 비밀번호 등)
5. CSRF 보호 여부
6. 입력값 검증 누락
7. 안전하지 않은 직렬화/역직렬화
8. 경로 순회(path traversal) 취약점
```

```
src/api/ 디렉토리의 모든 엔드포인트를 검토해서:
1. 입력값 검증이 누락된 곳
2. 인증 미들웨어가 빠진 라우트
3. Rate limiting이 없는 민감 API
를 찾아줘.
```

```
이 인증 시스템의 보안을 검토해줘:
- JWT 토큰 처리 방식
- 비밀번호 해싱 알고리즘
- 세션 관리 로직
- CORS 설정
```

```
환경변수와 시크릿 관리를 검토해줘.
하드코딩된 API 키나 비밀번호가 있는지,
.env 파일이 .gitignore에 포함되어 있는지,
민감 정보가 로그에 출력되는지 확인해줘.
```

### Claude의 예상 동작

- 인증/인가 관련 코드를 집중적으로 검토
- 사용자 입력을 처리하는 모든 경로를 추적
- 환경 변수와 설정 파일에서 민감 정보 노출 여부 확인
- 데이터베이스 쿼리에서 파라미터 바인딩 사용 여부 검증
- 알려진 취약 패턴(eval, innerHTML 직접 사용 등) 탐지
- 발견된 취약점을 심각도별로 분류하여 보고

### 실전 활용 팁

```
# 특정 영역 집중 검토
"API 인증 미들웨어의 보안을 검토해줘. 특히 JWT 토큰 검증 로직을 확인해줘"

# OWASP Top 10 기준 검토
"OWASP Top 10 기준으로 이 웹 애플리케이션의 보안을 검토해줘"
```

> **팁**: 보안 감사는 범위를 좁혀서 반복 실행하는 것이 효과적입니다. 먼저 `인증 관련 코드만 보안 검토해줘`, 그 다음 `API 입력값 검증을 검토해줘` 순서로 진행하세요.

## 6. 성능 분석

코드에서 성능 병목이 될 수 있는 부분을 식별합니다.

### 기본 프롬프트

```
성능 병목을 찾아줘
```

### 상세 프롬프트 예시

```
이 프로젝트의 성능을 다음 관점에서 분석해줘:

1. N+1 쿼리 문제가 있는 곳
2. 불필요한 리렌더링이 발생하는 컴포넌트
3. 메모이제이션이 필요한 연산
4. 비효율적인 알고리즘 (O(n²) 이상)
5. 캐싱이 필요한 API 호출
6. 큰 데이터를 한 번에 로드하는 곳
7. 병렬 처리가 가능한데 순차 처리하는 곳
```

```
데이터베이스 관련 코드를 분석해서 성능 이슈를 찾아줘:
- N+1 쿼리 패턴
- 인덱스 없이 자주 조회되는 컬럼
- 불필요한 전체 테이블 스캔
- 트랜잭션이 너무 길게 유지되는 곳
```

```
이 React 앱의 렌더링 성능을 분석해줘:
- 불필요한 리렌더링이 발생하는 컴포넌트
- 메모이제이션이 필요한 곳
- 번들 스플리팅이 가능한 라우트
- 큰 리스트에서 가상 스크롤이 필요한 곳
```

```
이 API 서버의 응답 시간을 개선할 수 있는 부분을 찾아줘:
- 캐싱이 가능한 응답
- 병렬화할 수 있는 순차 호출
- 불필요한 데이터 변환
- 최적화할 수 있는 알고리즘
```

### Claude의 예상 동작

- 데이터베이스 쿼리 패턴을 분석하여 N+1 문제 식별
- 반복문 내의 비효율적인 로직을 찾기
- React 컴포넌트의 불필요한 리렌더링 패턴 검출
- 대용량 데이터 처리 로직의 최적화 포인트 제안
- 구체적인 최적화 코드와 예상 효과를 제시

### 실전 활용 팁

```
# 데이터베이스 성능
"이 Prisma 쿼리들을 분석해서 N+1 문제가 있는 곳을 찾고 최적화 방법을 알려줘"

# API 응답 시간 개선
"이 API 엔드포인트의 응답이 느려. 코드를 분석해서 병목 지점을 찾아줘"

"이 React 컴포넌트가 느리게 렌더링돼. useMemo, useCallback 등으로 최적화할 부분을 찾아줘"

# 빌드 성능
"빌드 시간이 너무 오래 걸려. webpack/vite 설정을 분석해서 최적화 방법을 제안해줘"
```

> **팁**: 성능 분석 시 실제 병목 데이터(프로파일러 결과, 느린 쿼리 로그 등)가 있다면 함께 제공하세요. 추측 기반 최적화보다 데이터 기반 최적화가 훨씬 효과적입니다.

## 분석 워크플로우 조합 활용

실무에서는 여러 분석을 조합하여 사용하는 경우가 많습니다.

### 프로젝트 온보딩 시

```
새로 합류한 프로젝트야. 다음 순서로 분석해줘:
1. 전체 프로젝트 구조와 기술 스택
2. 핵심 비즈니스 로직의 위치와 흐름
3. 테스트 커버리지가 낮은 핵심 모듈
4. 현재 알려진 기술 부채나 TODO 주석
```

### 프로덕션 이슈 대응

```
1단계: 이 에러 로그를 분석해서 원인을 추적해줘: [에러 로그 붙여넣기]
2단계: 관련 코드의 최근 변경 이력을 확인해줘
3단계: 같은 유형의 에러가 발생할 수 있는 다른 코드도 찾아줘
4단계: 수정 코드와 함께 재발 방지를 위한 테스트 코드를 작성해줘
```

### 코드 리뷰 전 사전 분석

```
이 PR의 변경사항을 다음 관점에서 분석해줘:
1. 아키텍처 패턴 준수 여부
2. 보안 취약점
3. 성능 영향
4. 테스트 충분성
```

### 정기 코드 품질 점검

```
1단계: 보안 취약점을 검토해줘
2단계: 사용하지 않는 의존성과 코드를 찾아줘
3단계: 성능 병목을 찾아줘
4단계: 발견된 이슈를 심각도별로 정리하고 개선 로드맵을 제안해줘
```

### 기술 부채 평가

```
이 프로젝트의 기술 부채를 평가해줘:
1. 코드 복잡도가 높은 파일 (순환 복잡도 기준)
2. 중복 코드
3. 오래된 의존성
4. TODO/FIXME/HACK 주석 현황
5. 테스트가 없는 핵심 로직
```

## 핵심 정리

| 분석 유형 | 핵심 프롬프트 | 주요 확인 사항 | 활용 시점 |
|-----------|--------------|--------------|----------|
| 레거시 코드 이해 | "프로젝트 구조를 설명해줘" | 디렉토리 구조, 진입점, 데이터 흐름 | 프로젝트 온보딩 |
| 아키텍처 분석 | "아키텍처 패턴을 분석해줘" | 레이어 분리, 의존성 방향, 설계 패턴 | 설계 리뷰, 리팩토링 전 |
| 버그 원인 분석 | "에러 원인을 추적해줘" | 스택 트레이스, 데이터 흐름, 재현 조건 | 버그 수정 |
| 의존성 분석 | "사용하지 않는 의존성을 찾아줘" | 미사용 패키지, 중복, deprecated | 정기 점검, 업그레이드 |
| 보안 감사 | "보안 취약점을 검토해줘" | OWASP Top 10, 입력 검증, 인증 | 릴리스 전, 정기 점검 |
| 성능 분석 | "성능 병목을 찾아줘" | N+1 쿼리, 불필요 렌더링, 캐싱 | 성능 이슈 대응 |

> **팁**: 분석 요청 시 구체적인 컨텍스트(에러 메시지, 발생 조건, 관심 영역)를 함께 제공하면 Claude가 훨씬 정확한 분석 결과를 제공합니다.
