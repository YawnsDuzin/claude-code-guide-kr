## 프롬프트 구조화 6대 원칙

### 1) 역할 (Role)

에이전트의 기본 역할을 고정합니다.

```
너는 이 프로젝트의 시니어 개발자다. CLAUDE.md와 docs/ 아래 문서들이
단일 진실 공급원(single source of truth)이다.
```

### 2) 참조 (Context)

어떤 문서/파일을 먼저 읽어야 하는지 **경로로** 명시합니다.

```
먼저 다음 파일을 읽어:
- CLAUDE.md
- docs/PRD.md (§2.3 결제)
- docs/architecture.md (§결제 플로우)
- src/services/payment/*
```

### 3) 목적 (Intent)

"왜 하는지"를 한 줄로. 이게 있어야 에이전트가 세부 판단을 합니다.

### 4) 범위 (Scope)

건드려도 되는 것 / 안 되는 것을 명시.

### 5) 완료 기준 (Done Criteria)

체크박스 형태로 3~7개.

### 6) 제약 (Constraints)

성능, 보안, 호환성, 금지된 패턴 등.

## 공통 안티패턴

| 하지 마라 | 이유 |
|----------|------|
| "이거 좀 봐줘" | 범위가 무한. 에이전트는 소설을 씀. |
| "최대한 예쁘게" | 판단 기준이 없음. |
| "이런 거 알지?" | 에이전트 메모리는 세션마다 리셋. |
| 파일 경로 없이 "그 컴포넌트" | 추측 → 엉뚱한 파일 수정. |
| 완료 기준 없음 | 언제 끝났는지 판단 불가. |

## 프롬프트를 버전 관리하라

이 폴더 자체가 이미 git에 있습니다. 프로젝트에서 쓰면서 개선된 프롬프트는 **되돌려서 이 폴더에 PR**을 올리세요. 다음 프로젝트에서 재사용됩니다.
