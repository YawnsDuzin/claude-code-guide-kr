# 06. 코드 리뷰 프롬프트

> 에이전트를 **2차 리뷰어**로 쓰는 패턴. 1차 리뷰는 사람이, 2차 리뷰는 AI가.

## 두 가지 사용 시나리오

1. **자기 코드 리뷰**: 커밋 전에 내 변경을 스스로 리뷰
2. **독립 2차 의견**: 이미 동료가 리뷰한 PR에 다른 시점에서 한 번 더

---

## 템플릿 A: 자기 코드 리뷰 (Pre-commit)

```markdown
## 역할
너는 이 프로젝트의 **엄격한 시니어 리뷰어**다.
칭찬은 하지 마라. 개선점만 찾아라.

## 참조
- CLAUDE.md
- docs/architecture.md

## 입력
다음은 내가 방금 커밋하려는 변경이다.

```diff
<git diff 붙여넣기>
```

## 체크 항목
다음 7가지를 순서대로 점검하라.

1. **정확성**: 엣지 케이스 / null / 에러 경로 누락
2. **CLAUDE.md 규칙**: 금지 패턴 (any, console.log 등) 침범 여부
3. **테스트**: 이 변경을 커버하는 테스트가 있는가? 없다면 어떤 테스트가 필요한가?
4. **성능**: N+1, 불필요한 재렌더, 큰 객체 복사 등
5. **보안**: 사용자 입력 검증, SQL/XSS, 권한 체크
6. **가독성**: 이름, 함수 크기, 책임 분리
7. **문서 동기화**: PRD/erd/architecture 업데이트 필요 여부

## 출력 형식
- 각 항목: "OK" 또는 "문제: <한 줄>" + (있으면) 수정 제안 1개
- 마지막에 **머지 가능/불가** 판정
- 불가일 경우: 반드시 고쳐야 할 항목 목록

절대 칭찬하지 마라. "좋네요" 금지.
```

---

## 템플릿 B: 독립 2차 의견

동료가 이미 리뷰한 PR에 대해 다른 시점을 얻고 싶을 때.

```markdown
## 역할
너는 이 프로젝트 외부의 **독립 시니어 리뷰어**다.
기존 리뷰어의 의견을 모른다. 제로 베이스에서 판단하라.

## 참조
- CLAUDE.md
- docs/architecture.md
- PR 설명: <요약 붙여넣기>
- 변경 diff: <붙여넣기>

## 지시
나는 이 변경이 "안전하게 머지 가능한지"에 대한 **독립 의견**이 필요하다.

다음 순서로 답하라.

1. 이 변경이 해결하려는 문제 재서술 (내가 말한 것과 일치하는지 검증)
2. 선택한 접근이 최선인가? 대안 1~2개
3. 위험 요소 Top 3 (롤백 난이도 / 데이터 손실 / 성능 회귀 순)
4. 내가 놓쳤을 법한 것 (테스트, 문서, 관측성, 보안)
5. 머지 권장 여부: Yes / Yes with changes / No
```

---

## 왜 "독립" 시점이 중요한가

동료 리뷰어가 같은 팀이면 암묵 가정(implicit assumption)을 공유합니다. 그래서 **"당연히 X일 것"** 이라는 가정을 둘 다 검증하지 않고 넘어갑니다.

에이전트는 맥락을 강제로 문서에서만 얻기 때문에, 암묵 가정을 드러내주는 경우가 많습니다.

---

## 안티패턴

| 하지 마라 | 왜 |
|----------|----|
| "괜찮은지 봐줘" | 판단 기준 없음. 에이전트가 "괜찮아요" 로 답함. |
| 칭찬 허용 | 에이전트 칭찬은 쓸모없는 노이즈 |
| diff 없이 파일 경로만 | 현재 상태를 모르고 리뷰 |
| 완료 조건 없음 | "머지 가능/불가" 판정 없이 중립적 코멘트만 나옴 |
