# 00. 범용 프롬프트 템플릿 (Universal Template)

> 모든 작업 유형의 뿌리가 되는 6-section 구조. 어떤 에이전트에도 같은 구조로 던진다.

## 템플릿 (복사해서 사용)

```markdown
## 1. 역할 (Role)
너는 이 프로젝트의 시니어 개발자다.
`CLAUDE.md`와 `docs/` 하위 문서가 단일 진실 공급원이다.
문서에 없는 사실을 추정해야 할 경우, 먼저 질문하라.

## 2. 참조 (Context)
먼저 다음 파일을 읽고 시작하라.
- CLAUDE.md
- docs/PRD.md  §<해당 섹션>
- docs/architecture.md  §<해당 섹션>
- docs/erd.md  (스키마 관련일 때만)
- <관련 소스 파일 경로들>

## 3. 목적 (Intent)
<한 문장: 이 작업을 왜 하는가 — 유저 가치 또는 기술 이유>

## 4. 범위 (Scope)
### 포함
- <건드려도 되는 파일/모듈>
- <건드려도 되는 동작>

### 제외 (Out of Scope)
- <건드리지 말아야 할 파일/모듈>
- <이번 PR에서 하지 말 것>

## 5. 완료 기준 (Done Criteria)
- [ ] <검증 가능한 조건 1>
- [ ] <검증 가능한 조건 2>
- [ ] <검증 가능한 조건 3>
- [ ] 관련 문서(PRD/erd/architecture) 동기화
- [ ] 기존 테스트 모두 통과
- [ ] 새 로직에 대한 테스트 추가

## 6. 제약 (Constraints)
- 금지: <예: any 타입, console.log, 신규 의존성>
- 유지: <예: 공개 API 시그니처, 에러 코드>
- 성능: <예: p95 < 200ms>
- 보안: <예: 사용자 입력 이스케이프>

## 7. 출력 형식 (Output Format)
먼저 다음 순서로 답하라.
1. 이해한 내용 요약 (2~3줄)
2. 변경할 파일 목록 + 각 파일의 변경 의도
3. 내가 승인하면 → 코드 작성
4. 마지막에 완료 기준 체크리스트 체크 결과
```

---

## 왜 7개 섹션인가

| 섹션 | 없으면 생기는 문제 |
|------|------------------|
| 1. 역할 | 에이전트가 어떤 톤/수준으로 답할지 흔들림 |
| 2. 참조 | 문서를 안 읽고 추측 → 환각 |
| 3. 목적 | 세부 판단 실수 (트레이드오프 잘못 선택) |
| 4. 범위 | 스코프 확장 |
| 5. 완료 기준 | 무한 루프 / "다 됐어요" 거짓말 |
| 6. 제약 | 금지 패턴 침범 |
| 7. 출력 형식 | 코드만 훅 쏟아내서 리뷰 불가 |

---

## 축약 버전 (작은 작업용)

1~2파일 수준의 작은 작업에선 섹션을 압축해도 됩니다. 대신 **삭제하지 말고 한 줄로 줄이세요**.

```markdown
## 목적
<한 줄>

## 참조
- <파일 1>
- <파일 2>

## 완료 기준
- [ ] <조건 1>
- [ ] <조건 2>

## 제약
<한 줄>
```

---

## 7번 "출력 형식"이 중요한 이유

에이전트가 바로 코드를 쏟아내면, 잘못된 파일에 잘못된 변경을 쓴 뒤에야 알게 됩니다.
**"변경 계획"을 먼저 말하게 하면** 코드 생성 전에 잘못을 잡을 수 있습니다.

이 한 단계로 실패율이 체감상 절반으로 줍니다.
