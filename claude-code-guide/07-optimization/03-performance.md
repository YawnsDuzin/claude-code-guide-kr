# 성능 최적화 가이드

> Claude Code의 응답 속도와 효율성을 높이는 실전 팁을 다룹니다.

---

## 1. 모델 선택 전략

Claude Code에서는 작업 성격에 따라 모델을 선택할 수 있습니다.

| 모델 | 특징 | 적합한 작업 |
|------|------|------------|
| **Opus** | 가장 강력, 복잡한 추론 | 아키텍처 설계, 대규모 리팩토링, 복잡한 버그 |
| **Sonnet** | 균형잡힌 성능/속도 | 일반 코딩, 기능 구현, 코드 리뷰 |
| **Haiku** | 가장 빠름, 간단한 작업 | 간단한 수정, 포맷팅, 파일 검색 |

### 모델 변경 방법

```bash
# 세션 시작 시
claude --model sonnet

# 세션 중 변경
/model sonnet
```

**팁:** 탐색/분석은 Haiku, 구현은 Sonnet, 복잡한 설계는 Opus로 조합하면 비용 대비 효율이 좋습니다.

## 2. 컨텍스트 관리 최적화

### /compact 전략적 사용

컨텍스트가 커질수록 응답이 느려지고 비용이 증가합니다.

```
# 대화가 길어졌을 때 요약 압축
/compact

# 특정 주제로 압축
/compact 인증 시스템 구현에 집중
```

**사용 타이밍:**
- 대화가 20턴 이상 진행되었을 때
- 주제가 바뀌었을 때
- "context window 부족" 경고가 나올 때

### 세션 분리

하나의 긴 세션보다 여러 짧은 세션이 효율적입니다:

```bash
# 작업 1: 데이터 모델 설계
claude "User, Product, Order 모델을 Prisma 스키마로 설계해줘"

# 작업 2: API 구현 (새 세션)
claude "prisma/schema.prisma 기반으로 User CRUD API를 구현해줘"

# 작업 3: 테스트 (새 세션)
claude "src/routes/user.ts의 테스트를 작성해줘"
```

## 3. 도구 사용 최적화

### Glob/Grep 우선 사용

파일 검색에 Bash 대신 전용 도구를 사용하면 더 빠릅니다:

```
# 느림 (Bash)
"find . -name '*.ts' -exec grep 'export class' {} \; 실행해줘"

# 빠름 (전용 도구 유도)
"프로젝트에서 export class가 포함된 TypeScript 파일을 찾아줘"
```

Claude는 자동으로 적절한 도구를 선택하지만, 명확한 지시가 도움됩니다.

### 파일 읽기 최적화

```
# 비효율적: 전체 파일 읽기
"src/services/orderService.ts 전체를 읽어줘"

# 효율적: 필요한 부분만
"src/services/orderService.ts의 processOrder 함수만 읽어줘"
```

## 4. 서브에이전트 병렬화

독립적인 작업은 서브에이전트로 병렬 실행하면 빠릅니다:

### 병렬 처리가 효과적인 경우

```
다음 세 가지를 동시에 분석해줘:
1. src/api/ 디렉토리의 엔드포인트 목록
2. src/models/ 디렉토리의 데이터 모델 구조
3. package.json의 의존성 분석
```

Claude가 자동으로 3개의 서브에이전트를 실행합니다.

### 병렬 처리가 비효율적인 경우

순차적으로 의존하는 작업:
```
# 이건 순차적으로 진행해야 함
1. 먼저 DB 스키마 설계
2. 스키마 기반으로 API 구현
3. API 기반으로 테스트 작성
```

## 5. 배치 작업 최적화

### 여러 파일 동시 수정

```
# 비효율적: 하나씩 요청
"UserCard.tsx를 수정해줘" → "ProductCard.tsx도 수정해줘" → ...

# 효율적: 한 번에 요청
"components/ 디렉토리의 모든 Card 컴포넌트에서:
- className prop 추가
- 기본 스타일을 Tailwind로 변경
- onClick 핸들러 타입을 MouseEventHandler로 통일"
```

### 일괄 리팩토링

```
src/api/ 디렉토리의 모든 라우트 핸들러에서:
- try-catch를 asyncHandler 래퍼로 교체
- res.json() 호출을 sendResponse 유틸로 통일
- 에러 응답을 AppError 클래스로 표준화
```

## 6. 네트워크 고려사항

| 상황 | 최적화 방법 |
|------|------------|
| 느린 네트워크 | 작은 단위로 요청, 응답 대기 시간 고려 |
| API 레이트 리밋 | 요청 간격 조절, 큰 작업은 나누어 실행 |
| 타임아웃 | 복잡한 작업은 단계를 나누어 실행 |

## 7. CLAUDE.md로 반복 줄이기

매 세션마다 반복하는 설명은 CLAUDE.md에 포함시키세요:

```markdown
# 자주 사용하는 컨텍스트

## 에러 핸들링 패턴
모든 서비스는 AppError를 throw하고,
글로벌 에러 핸들러(src/middleware/errorHandler.ts)에서 처리.

## 테스트 패턴
- 파일명: *.test.ts
- 위치: 대상 파일과 같은 디렉토리
- Mock: __mocks__/ 디렉토리
```

이렇게 하면 매번 "우리 프로젝트는 이런 패턴을 써..." 라고 설명할 필요가 없습니다.
