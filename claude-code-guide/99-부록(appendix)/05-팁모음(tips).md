# Claude Code 알아두면 좋은 팁 30선

실전에서 바로 활용할 수 있는 Claude Code 팁을 모았습니다.

---

## 기본 사용 팁

### 1. 구체적으로 지시하기
```
# 나쁜 예
"코드 고쳐줘"

# 좋은 예
"src/utils/auth.ts의 validateToken 함수에서
만료된 토큰을 처리하지 못하는 버그를 수정해줘"
```

### 2. 파일 경로 명시하기
```
"src/components/Header.tsx에서 네비게이션 메뉴를 추가해줘"
```
경로를 지정하면 Claude가 파일을 찾는 시간을 절약합니다.

### 3. 한 번에 하나씩 요청하기
여러 작업을 한꺼번에 요청하면 놓치는 부분이 생깁니다. 작업을 나눠서 요청하세요.

### 4. `/commit` 적극 활용하기
```
# 변경 후 바로 커밋
/commit

# Claude가 diff를 분석해서 적절한 커밋 메시지를 자동 생성합니다
```

### 5. 작업 전에 현황 파악 시키기
```
"이 프로젝트의 구조를 먼저 파악해줘"
"package.json을 확인하고 사용 중인 라이브러리를 알려줘"
```

## 세션 관리 팁

### 6. 새 작업은 새 세션에서
```
/clear
"새로운 기능을 개발할 거야. 사용자 프로필 페이지를 만들어줘"
```
이전 작업의 컨텍스트가 새 작업을 방해할 수 있습니다.

### 7. `/compact`으로 대화 압축
대화가 길어지면 `/compact`로 이전 대화를 요약합니다. 토큰 절약 + 성능 유지.

### 8. 이전 세션 이어가기
```bash
# 마지막 세션 이어서 시작
claude -c

# 특정 세션 이어가기
claude -r
```

### 9. 컨텍스트 상태 확인
세션이 느려지면 `/cost`로 토큰 사용량을 확인하세요.

### 10. 작업 완료 후 항상 커밋
```
"작업 완료했으면 변경사항을 커밋해줘"
```
커밋하지 않으면 다음 세션에서 변경사항을 잃을 수 있습니다.

## 코딩 팁

### 11. 테스트 먼저 실행시키기
```
"먼저 기존 테스트를 실행해서 현재 상태를 확인해줘"
```
변경 전 기준점을 만들어두면 문제를 쉽게 찾을 수 있습니다.

### 12. 에러 메시지 그대로 붙여넣기
```
"이 에러가 발생했어:
TypeError: Cannot read properties of undefined (reading 'map')
at UserList.tsx:42"
```
에러 메시지를 그대로 보여주면 Claude가 정확히 진단합니다.

### 13. 기존 패턴을 따르게 하기
```
"src/api/userApi.ts를 참고해서 같은 패턴으로 productApi.ts를 만들어줘"
```

### 14. 단계별로 진행 요청
```
"다음 순서대로 진행해줘:
1. DB 모델 생성
2. API 엔드포인트 추가
3. 테스트 작성
4. 각 단계마다 커밋"
```

### 15. 변경 범위 제한하기
```
"이 함수만 수정해줘. 다른 파일은 건드리지 마"
```
Claude가 불필요하게 다른 코드를 수정하는 것을 방지합니다.

## CLAUDE.md 팁

### 16. 핵심만 간결하게
50-100줄이 적정. 너무 길면 컨텍스트를 낭비합니다.

### 17. 테스트/빌드 명령어 반드시 포함
```markdown
# 명령어
- 테스트: npm test
- 빌드: npm run build
- 린트: npm run lint
```
Claude가 자동으로 검증할 수 있습니다.

### 18. 하지 말아야 할 것 명시
```markdown
# 금지 사항
- console.log를 커밋하지 않는다
- any 타입을 사용하지 않는다
- CSS 파일을 생성하지 않는다 (Tailwind 사용)
```

### 19. 하위 폴더별 CLAUDE.md 활용
```
src/
├── CLAUDE.md              # 전체 소스 규칙
├── components/
│   └── CLAUDE.md          # 컴포넌트 규칙
└── api/
    └── CLAUDE.md          # API 규칙
```

### 20. Git에 커밋하여 팀 공유
CLAUDE.md를 Git에 포함시키면 팀 전체가 같은 규칙을 사용합니다.

## 디버깅 팁

### 21. 로그/에러 출력 전체를 보여주기
```
"npm run build를 실행하고 에러가 있으면 보여줘"
```

### 22. 가설을 같이 제시하기
```
"로그인 후 리다이렉트가 안 돼. 
세션 토큰이 저장되지 않는 것 같은데 확인해줘"
```

### 23. 최소 재현 요청
```
"이 버그의 최소 재현 코드를 만들어줘"
```

### 24. git blame 활용
```
"이 버그가 언제부터 생겼는지 git blame으로 확인해줘"
```

### 25. 비교 디버깅
```
"working-branch에서는 되는데 이 브랜치에서는 안 돼.
두 브랜치의 차이를 비교해줘"
```

## 생산성 팁

### 26. 비대화형 모드로 자동화
```bash
# 파이프라인에서 코드 리뷰
claude -p "$(git diff HEAD~1)" --append-system-prompt "한국어로 코드 리뷰해줘"

# 파일 분석
claude -p "src/api/를 분석하고 개선점을 알려줘"
```

### 27. 여러 파일 동시 작업
```
"다음 파일들을 모두 수정해줘:
- src/types/user.ts에 email 필드 추가
- src/api/userApi.ts에 이메일 검증 추가
- src/components/UserForm.tsx에 이메일 입력 추가"
```

### 28. 프로젝트 온보딩에 활용
```
"이 프로젝트에 새로 합류했어. 전체 구조와
주요 파일을 설명해줘. 개발 시작하려면 뭘 해야 하는지도 알려줘"
```

### 29. 리뷰 자동화
```
"내 변경사항을 PR 올리기 전에 셀프 리뷰해줘.
console.log, TODO, 하드코딩된 값이 있는지 확인해줘"
```

### 30. 문서화 자동화
```
"이 프로젝트의 API 엔드포인트를 분석해서
API 문서를 마크다운으로 작성해줘"
```

## 빠른 참조: 슬래시 명령어

| 명령어 | 설명 |
|--------|------|
| `/commit` | 변경사항 자동 커밋 |
| `/compact` | 대화 압축 (토큰 절약) |
| `/clear` | 대화 초기화 |
| `/cost` | 토큰 사용량 확인 |
| `/help` | 도움말 |
| `/init` | CLAUDE.md 생성 |
| `/review` | 코드 리뷰 |
| `/bug` | 버그 리포트 생성 |
