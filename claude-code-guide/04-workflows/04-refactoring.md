# 리팩토링 워크플로우

Claude Code를 활용하여 안전하고 체계적으로 코드를 리팩토링하는 방법을 소개합니다. 코드 스멜 식별부터 대규모 마이그레이션까지 실무에서 바로 적용할 수 있는 워크플로우를 다룹니다.

---

## 1. 코드 스멜(Code Smell) 식별

리팩토링이 필요한 부분을 체계적으로 찾습니다.

### 기본 프롬프트

```
이 파일에서 코드 스멜을 찾아줘: src/services/orderService.ts
```

### 상세 프롬프트 예시

```
이 프로젝트에서 다음 코드 스멜을 찾아줘:

1. 너무 긴 함수 (30줄 이상)
2. 매개변수가 많은 함수 (4개 이상)
3. 중복 코드
4. 깊은 중첩 (3단계 이상)
5. God 클래스 / God 함수
6. 매직 넘버/문자열
7. 주석으로 설명이 필요한 복잡한 로직
8. 일관성 없는 에러 처리

각각에 대해 위치와 개선 방안을 알려줘.
```

### Claude의 예상 동작

- 파일 또는 디렉토리를 스캔하여 코드 스멜 패턴을 식별
- 각 문제의 심각도를 평가 (높음/중간/낮음)
- 구체적인 리팩토링 방법을 제안
- 우선순위를 권장

### 실전 활용 팁

```
# 특정 디렉토리 전체 검사
"src/services/ 디렉토리의 모든 파일에서 코드 스멜을 찾아줘"

# 복잡도 기준 검사
"순환 복잡도(cyclomatic complexity)가 높은 함수를 찾아줘"

# 중복 코드 집중 검사
"프로젝트 전체에서 중복되는 코드 패턴을 찾아줘"
```

## 2. 함수 추출 (Extract Function)

긴 함수를 의미 있는 단위로 분리합니다.

### 프롬프트 예시

```
src/controllers/orderController.ts의 createOrder 함수가 너무 길어.
다음 기준으로 함수를 분리해줘:

1. 유효성 검사 로직 → validateOrderInput()
2. 재고 확인 로직 → checkInventory()
3. 가격 계산 로직 → calculateTotal()
4. 주문 생성 로직 → saveOrder()
5. 알림 발송 로직 → sendOrderNotification()

각 함수는:
- 단일 책임을 가지도록
- 명확한 입력/출력 타입을 가지도록
- 테스트하기 쉽도록 순수 함수로 만들 수 있으면 순수 함수로
```

### Claude의 예상 동작

- 기존 함수의 로직을 분석하여 의미 단위로 분리
- 추출된 함수 간의 데이터 흐름을 정리
- 타입 정의를 함께 생성
- 원래 함수를 추출된 함수들의 조합으로 재구성
- 기존 동작이 변경되지 않는지 확인

### 핵심 팁

```
# 단계적 추출
"먼저 가장 독립적인 로직부터 추출하고, 하나씩 확인하면서 진행하자"

# 테스트 먼저
"리팩토링 전에 현재 동작을 보장하는 테스트를 먼저 작성해줘"
```

## 3. 클래스/모듈 재구조화

책임이 과도하게 집중된 모듈을 분리합니다.

### 프롬프트 예시

```
src/services/userService.ts가 800줄이 넘어.
다음과 같이 분리해줘:

1. 인증 관련 → authService.ts
   - login, logout, refreshToken, validateSession
2. 프로필 관련 → profileService.ts
   - getProfile, updateProfile, uploadAvatar
3. 권한 관련 → permissionService.ts
   - checkPermission, assignRole, revokeRole
4. 알림 설정 → notificationPreferenceService.ts
   - getPreferences, updatePreferences

분리 후:
- 기존 import 경로를 모두 업데이트
- 인덱스 파일(index.ts)로 하위 호환성 유지
- 순환 의존이 생기지 않도록 주의
```

### Claude의 예상 동작

- 기존 파일의 함수/메서드를 분류
- 의존성 관계를 분석하여 순환 참조를 방지
- 새 파일들을 생성하고 코드를 이동
- 프로젝트 전체에서 import 경로를 업데이트
- 하위 호환성을 위한 re-export 설정

### 핵심 팁

```
# 의존성 확인 먼저
"이 파일을 분리하기 전에 다른 파일에서 어떻게 참조되고 있는지 확인해줘"

# 점진적 분리
"한 번에 다 분리하지 말고, 먼저 authService만 분리하고 테스트가 통과하는지 확인하자"
```

## 4. 네이밍 개선

변수명, 함수명, 파일명 등을 더 명확하게 변경합니다.

### 프롬프트 예시

```
src/utils/helpers.ts 파일의 함수명을 개선해줘:

현재:
- process() → 무엇을 처리하는지 불분명
- getData() → 어떤 데이터인지 불분명
- check() → 무엇을 체크하는지 불분명
- handle() → 무엇을 핸들링하는지 불분명
- calc() → 무엇을 계산하는지 불분명

함수의 실제 동작을 분석하여 의도가 명확한 이름을 제안해줘.
프로젝트 전체에서 사용되는 곳도 함께 변경해줘.
```

### 일괄 네이밍 컨벤션 적용

```
이 프로젝트의 네이밍 컨벤션을 통일해줘:

규칙:
- 변수/함수: camelCase
- 클래스/인터페이스/타입: PascalCase
- 상수: UPPER_SNAKE_CASE
- 파일명: camelCase (컴포넌트는 PascalCase)
- 불리언 변수: is/has/should 접두사
- 이벤트 핸들러: handle 접두사 (handleClick, handleSubmit)

위반하는 곳을 찾아서 수정해줘.
```

### Claude의 예상 동작

- 함수의 실제 구현을 읽어 의도를 파악
- 프로젝트의 기존 네이밍 패턴을 고려
- 이름 변경 시 사용되는 모든 곳을 함께 수정
- 변경 전/후를 비교하여 보여줌

## 5. 죽은 코드(Dead Code) 제거

사용되지 않는 코드를 찾아서 제거합니다.

### 프롬프트 예시

```
이 프로젝트에서 사용되지 않는 코드를 찾아줘:

1. import되었지만 사용하지 않는 모듈
2. 정의되었지만 호출되지 않는 함수
3. export되었지만 다른 곳에서 import하지 않는 것
4. 주석 처리된 코드 블록
5. 도달할 수 없는 코드 (unreachable code)
6. 사용하지 않는 변수와 매개변수
7. 사용하지 않는 타입/인터페이스 정의

찾은 항목을 파일별로 정리하고, 안전하게 제거할 수 있는 것과
확인이 필요한 것을 구분해줘.
```

### Claude의 예상 동작

- 프로젝트 전체의 import/export 관계를 분석
- 함수 호출 그래프를 추적
- 동적 import나 리플렉션으로 사용되는 코드는 주의하여 표시
- 안전하게 제거할 수 있는 코드와 확인이 필요한 코드를 구분

### 핵심 팁

```
# 단계적 제거
"먼저 확실하게 사용되지 않는 것만 제거하고 빌드/테스트 확인 후 다음 단계로 진행하자"

# 동적 참조 주의
"리플렉션이나 동적 import로 사용될 수 있는 코드는 제거하지 말고 표시만 해줘"
```

## 6. 디자인 패턴 적용

적절한 디자인 패턴을 적용하여 코드 구조를 개선합니다.

### 프롬프트 예시

```
src/services/notificationService.ts를 보면 알림 타입별로 if-else가 길게 이어져 있어.
Strategy 패턴을 적용해서 리팩토링해줘.

현재 구조:
if (type === 'email') { ... }
else if (type === 'sms') { ... }
else if (type === 'push') { ... }
else if (type === 'slack') { ... }

패턴 적용 후:
- NotificationStrategy 인터페이스
- EmailStrategy, SmsStrategy, PushStrategy, SlackStrategy 구현체
- NotificationContext로 전략 선택
- 새 알림 타입 추가 시 기존 코드 수정 없이 확장 가능하도록
```

### 다른 패턴 적용 예시

```
# Observer 패턴
"주문 상태 변경 시 여러 서비스에 알림을 보내는 코드가 강결합되어 있어.
Observer 패턴으로 리팩토링해줘"

# Factory 패턴
"여러 곳에서 유사한 객체 생성 로직이 중복돼. Factory 패턴으로 통합해줘"

# Repository 패턴
"컨트롤러에서 직접 DB 쿼리를 실행하고 있어. Repository 패턴으로 분리해줘"

# Decorator 패턴
"여러 함수에 로깅, 캐싱, 유효성 검사 로직이 반복돼. Decorator 패턴으로 분리해줘"
```

### Claude의 예상 동작

- 현재 코드의 문제점을 진단
- 적절한 디자인 패턴을 선택하고 이유를 설명
- 패턴 적용 전/후의 구조를 비교
- 점진적으로 리팩토링하여 기존 동작을 유지

## 7. 대용량 파일 분할

너무 커진 파일을 논리적 단위로 분할합니다.

### 프롬프트 예시

```
src/utils/index.ts가 1500줄이야. 논리적으로 분할해줘.

분할 기준:
1. 관련 기능별로 그룹핑 (날짜, 문자열, 배열, 유효성검사 등)
2. 각 파일은 200줄 이하
3. 순환 의존이 생기지 않도록
4. 기존 import 경로 호환성 유지 (index.ts에서 re-export)

예상 결과:
src/utils/
├── index.ts          # re-export만
├── dateUtils.ts      # 날짜 관련
├── stringUtils.ts    # 문자열 관련
├── arrayUtils.ts     # 배열 관련
├── validationUtils.ts # 유효성 검사
├── formatUtils.ts    # 포맷팅
└── types.ts          # 공통 타입
```

### Claude의 예상 동작

- 파일 내 함수들의 관계를 분석
- 논리적 그룹을 식별
- 새 파일들로 코드를 분리
- `index.ts`에서 모든 것을 re-export하여 기존 코드 호환성 유지
- 프로젝트 전체의 import를 업데이트 (선택적)

## 8. 마이그레이션 (JavaScript → TypeScript 등)

기술 스택 전환을 체계적으로 수행합니다.

### JavaScript → TypeScript 마이그레이션

```
src/services/userService.js를 TypeScript로 변환해줘.

요구사항:
1. 모든 함수에 매개변수와 반환 타입 명시
2. any 사용 최소화 (불가피한 경우 주석으로 이유 설명)
3. 인터페이스/타입을 별도 파일로 분리
4. null/undefined 처리를 strict 모드에 맞게
5. 기존 JSDoc 주석의 타입 정보를 TypeScript 타입으로 변환
6. 기존 동작이 변경되지 않도록
```

### 점진적 마이그레이션 전략

```
이 프로젝트를 JavaScript에서 TypeScript로 점진적으로 마이그레이션하고 싶어.
다음 계획을 세워줘:

1. tsconfig.json 설정 (allowJs: true로 시작)
2. 마이그레이션 우선순위 (의존성이 적은 파일부터)
3. 공통 타입 정의 먼저 생성
4. 파일별 변환 순서
5. 각 단계에서 테스트 검증 방법
```

### 다른 마이그레이션 예시

```
# CSS → Tailwind
"이 컴포넌트의 CSS를 Tailwind 클래스로 변환해줘"

# Class Component → Function Component
"이 React 클래스 컴포넌트를 함수형 컴포넌트 + Hooks로 변환해줘"

# REST → GraphQL
"이 REST 엔드포인트를 GraphQL 스키마와 리졸버로 변환해줘"

# CommonJS → ES Modules
"require/module.exports를 import/export로 변환해줘"

# Options API → Composition API (Vue)
"이 Vue 2 Options API 컴포넌트를 Vue 3 Composition API로 변환해줘"
```

### Claude의 예상 동작

- 기존 코드의 동작을 정확히 보존
- 새 기술 스택의 모범 사례를 적용
- 타입 추론이 가능한 곳은 명시적 타입 대신 추론 활용
- 변환 과정에서 발견된 잠재적 버그를 보고

## 9. 테스트와 함께하는 안전한 리팩토링

테스트를 활용하여 리팩토링의 안전성을 보장합니다.

### 워크플로우

```
# Step 1: 현재 상태 테스트 작성
"이 함수의 현재 동작을 그대로 보존하는 테스트를 먼저 작성해줘.
모든 분기와 엣지 케이스를 커버해줘."

# Step 2: 테스트 통과 확인
"테스트를 실행해서 모두 통과하는지 확인해줘"

# Step 3: 리팩토링 수행
"테스트가 통과하니까, 이제 이 함수를 리팩토링해줘.
[리팩토링 내용 설명]"

# Step 4: 테스트 재실행
"리팩토링 후 기존 테스트가 모두 통과하는지 확인해줘"

# Step 5: 테스트 업데이트 (필요시)
"리팩토링으로 인터페이스가 변경된 부분의 테스트를 업데이트해줘"
```

### 프롬프트 예시

```
src/services/pricingService.ts를 리팩토링하고 싶어.
현재 할인 계산 로직이 하나의 거대한 함수에 있어.

안전하게 진행하기 위해:
1. 먼저 현재 동작을 보장하는 테스트를 작성
2. 테스트 통과 확인
3. 함수를 작은 단위로 분리
4. 각 분리 후 테스트 재실행
5. 최종 통과 확인

이 순서로 진행해줘.
```

### Claude의 예상 동작

1. 기존 함수의 입출력을 분석하여 테스트 케이스 생성
2. 테스트 실행으로 현재 동작 확인
3. 테스트가 통과하는 상태를 유지하며 점진적 리팩토링
4. 각 변경마다 테스트 재실행
5. 최종적으로 모든 테스트 통과 확인

## 리팩토링 모범 사례

### 작게, 자주, 안전하게

```
# 나쁜 방식
"이 서비스 전체를 리팩토링해줘" → 변경 범위가 너무 넓어 문제 추적이 어려움

# 좋은 방식
"이 함수의 유효성 검사 로직만 분리해줘" → 작은 단위로 안전하게 변경
```

### 리팩토링 전 체크리스트

```
리팩토링 시작 전에 다음을 확인해줘:
1. 테스트 커버리지가 충분한지
2. 현재 모든 테스트가 통과하는지
3. 리팩토링 범위가 명확한지
4. 예상되는 영향 범위
```
