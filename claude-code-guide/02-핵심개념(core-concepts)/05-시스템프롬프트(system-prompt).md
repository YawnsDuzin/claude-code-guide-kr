# Claude Code의 시스템 프롬프트 이해하기

Claude Code가 내부적으로 어떤 지침을 따르는지, 그리고 이를 어떻게 커스터마이징할 수 있는지 알아봅니다.

---

## 시스템 프롬프트란?

시스템 프롬프트는 **AI의 행동 방식을 정의하는 초기 지시사항**입니다. 사용자가 보내는 메시지와 별도로, Claude가 "어떤 역할로, 어떤 규칙을 따라" 응답할지를 결정합니다.

```
┌─────────────────────────────────────┐
│         컨텍스트 윈도우              │
│                                     │
│  ① 시스템 프롬프트  ← Claude의 행동 규칙  │
│  ② CLAUDE.md       ← 프로젝트 설정     │
│  ③ 대화 기록        ← 사용자 메시지/응답  │
│  ④ 파일/실행 결과    ← 도구 사용 결과    │
└─────────────────────────────────────┘
```

---

## Claude Code의 기본 시스템 프롬프트 구조

Claude Code를 실행하면 자동으로 설정되는 시스템 프롬프트의 주요 내용입니다.

### 역할 정의

> "You are Claude Code, an interactive CLI tool that helps users with software engineering tasks."

- 소프트웨어 엔지니어링 전문 도구로 동작
- 터미널에서 파일 편집, 명령어 실행, 코드 분석 수행

### 도구 사용 규칙

| 규칙 | 설명 |
|------|------|
| 파일 읽기는 Read 도구 사용 | `cat`/`head` 대신 전용 도구 사용 |
| 파일 편집은 Edit 도구 사용 | `sed`/`awk` 대신 전용 도구 사용 |
| 검색은 Grep/Glob 사용 | `grep`/`find` 대신 전용 도구 사용 |
| Bash는 시스템 명령어 전용 | 빌드, 테스트 실행 등에만 사용 |

### 코딩 원칙

- **기존 코드를 먼저 읽고** 이해한 후 수정
- **요청한 것만 변경** (과도한 리팩토링, 주석 추가 금지)
- **기존 파일 편집 우선** (새 파일 생성 최소화)
- **보안 취약점 방지** (XSS, SQL Injection 등 OWASP Top 10)
- **불필요한 추상화 금지** (심플한 코드가 최선)

### 안전 규칙

- 위험한 작업(파일 삭제, force push 등) 전에 **반드시 사용자 확인**
- 민감 정보(.env, API 키) **커밋 방지**
- git commit은 **사용자가 요청할 때만** 실행
- `--no-verify` 등 안전 장치를 **우회하지 않음**

### 톤과 스타일

- 간결하고 직접적인 응답
- 이모지 사용 안 함 (요청 시에만)
- 코드 위치 표시: `file_path:line_number` 형식

### 환경 정보 (자동 포함)

시스템 프롬프트에 아래 정보가 자동으로 추가됩니다:

```
- 작업 디렉토리: /home/user/my-project
- Git 저장소 여부: true
- 플랫폼: linux / macOS
- 쉘: bash / zsh
- 모델: claude-sonnet-4-20250514
```

## 시스템 프롬프트 커스터마이징

### 방법 1: `--system-prompt` (완전 교체)

기본 시스템 프롬프트를 **완전히 대체**합니다. 비대화형(`-p`) 모드에서만 사용 가능합니다.

```bash
# 보안 리뷰 전용 모드
claude -p "src/ 디렉토리를 분석해줘" \
  --system-prompt "당신은 시니어 보안 전문가입니다.
코드에서 보안 취약점만 찾아서 심각도와 함께 보고하세요."

# 코드 리뷰 전용 모드
claude -p "$(git diff HEAD~1)" \
  --system-prompt "변경사항을 리뷰하세요.
버그, 성능 문제, 가독성을 중심으로 피드백하세요."
```

> **주의:** 기본 시스템 프롬프트가 완전히 대체되므로, 도구 사용 규칙이나 안전 지침이 사라집니다. 단순 분석 작업에 적합합니다.

### 방법 2: `--append-system-prompt` (추가)

기본 시스템 프롬프트를 **유지하면서** 추가 규칙을 덧붙입니다.

```bash
# 코딩 스타일 지정
claude -p "이 함수를 리팩토링해줘" \
  --append-system-prompt "변수명은 camelCase, 함수는 20줄 이내로 작성하세요."

# 초보자 친화적 설명
claude -p "이 코드를 설명해줘" \
  --append-system-prompt "초보 개발자에게 설명하듯 쉽게 답변하세요."

# 한국어 응답 강제
claude -p "Refactor this code" \
  --append-system-prompt "반드시 한국어로 응답하세요."
```

### 방법 3: CLAUDE.md (간접 커스터마이징)

시스템 프롬프트는 아니지만, 컨텍스트에 포함되어 **프로젝트별 규칙**을 설정합니다.

```markdown
# CLAUDE.md 예시
## 코딩 규칙
- TypeScript strict 모드 사용
- 함수형 컴포넌트만 사용 (class 컴포넌트 금지)
- 커밋 메시지는 Conventional Commits 형식

## 프로젝트 구조
- src/components/ - React 컴포넌트
- src/hooks/ - 커스텀 훅
- src/api/ - API 호출 함수
```

## 세 가지 방법 비교

| 구분 | 시스템 프롬프트 (`--system-prompt`) | 추가 프롬프트 (`--append-system-prompt`) | CLAUDE.md |
|------|------|------|------|
| **범위** | 전체 대체 | 기본 + 추가 | 프로젝트별 |
| **모드** | 비대화형만 | 비대화형만 | 대화형/비대화형 |
| **기본 규칙 유지** | X (사라짐) | O (유지) | O (유지) |
| **영구성** | 해당 실행만 | 해당 실행만 | 파일로 지속 |
| **용도** | 완전히 다른 역할 | 규칙 추가 | 프로젝트 설정 |

### 언제 무엇을 쓸까?

```
"Claude를 코드 리뷰어로만 쓰고 싶다"     → --system-prompt
"응답을 항상 한국어로 받고 싶다"           → --append-system-prompt
"프로젝트 코딩 규칙을 항상 따르게 하고 싶다"  → CLAUDE.md
```

## 활용 시나리오

### CI/CD 파이프라인에서 자동 코드 리뷰

```bash
# .github/workflows/review.yml
claude -p "$(git diff $BASE_SHA...$HEAD_SHA)" \
  --system-prompt "코드 리뷰어로서 변경사항을 분석하세요.
버그 가능성, 성능 이슈, 보안 취약점을 보고하세요.
JSON 형식으로 출력하세요." \
  --output-format json
```

### 문서 자동 생성

```bash
claude -p "src/api/ 디렉토리의 모든 API를 문서화해줘" \
  --append-system-prompt "OpenAPI 3.0 스펙에 맞춰 YAML로 출력하세요."
```

### 테스트 코드 생성

```bash
claude -p "src/utils/calculator.ts의 테스트를 작성해줘" \
  --append-system-prompt "Jest를 사용하고,
엣지 케이스를 포함해서 커버리지 90% 이상을 목표로 하세요."
```

## 주의사항

| 주의 | 설명 |
|------|------|
| **컨텍스트 소비** | 시스템 프롬프트가 길수록 남은 공간이 줄어듦 |
| **완전 대체 주의** | `--system-prompt`는 안전 규칙도 제거되므로 분석 작업에만 사용 |
| **프롬프트 캐싱** | 동일한 시스템 프롬프트는 자동 캐싱되어 비용 절감 |
| **CLAUDE.md 우선** | 일상적인 규칙은 CLAUDE.md에 넣는 것이 더 실용적 |
