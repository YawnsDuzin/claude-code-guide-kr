# 컨텍스트 윈도우 관리(Context Window Management)

Claude Code의 성능과 비용을 최적화하려면 **컨텍스트 윈도우(Context Window)**를 이해하고 효과적으로 관리하는 것이 중요합니다. 이 문서에서는 컨텍스트 윈도우의 개념부터 실전 관리 전략까지 알아봅니다.

---

## 컨텍스트 윈도우란?

컨텍스트 윈도우는 Claude가 **한 번에 처리할 수 있는 텍스트의 최대 크기**입니다. 대화 내용, 파일 내용, 도구 실행 결과 등 모든 정보가 이 윈도우 안에 들어가야 합니다.

### 크기

- **Claude Code의 컨텍스트 윈도우**: 약 **200K 토큰**
- 1 토큰 = 대략 영어 4글자 또는 한국어 1~2글자
- 200K 토큰 = 약 150,000 단어 = 일반 소설 2~3권 분량

### 컨텍스트 윈도우에 포함되는 것들

```
┌─────────────────────────────────────────┐
│          컨텍스트 윈도우 (200K)            │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │ 시스템 프롬프트 (Claude의 지침)     │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │ CLAUDE.md 내용                    │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │ 대화 기록                         │    │
│  │  - 사용자 메시지                    │    │
│  │  - Claude 응답                    │    │
│  │  - 도구 호출 및 결과                │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │ 읽은 파일 내용                     │    │
│  └─────────────────────────────────┘    │
│  ┌─────────────────────────────────┐    │
│  │ 명령어 실행 결과                    │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 왜 중요한가?

1. **응답 품질**: 컨텍스트가 가득 차면 오래된 정보가 압축되어 응답 품질이 떨어질 수 있음
2. **비용**: 토큰 사용량에 비례하여 비용이 발생 (더 많은 컨텍스트 = 더 높은 비용)
3. **속도**: 컨텍스트가 클수록 응답 생성 시간이 길어짐

## Claude Code의 자동 컨텍스트 관리

Claude Code는 컨텍스트 윈도우가 한계에 가까워지면 **자동으로 관리**합니다.

### 자동 압축(Auto-Compaction)

컨텍스트 사용량이 일정 수준(약 95%)에 도달하면, Claude Code가 자동으로 이전 대화를 **압축(compaction)**합니다.

#### 자동 압축의 동작 방식

1. 컨텍스트 사용량이 임계치에 도달
2. Claude가 이전 대화 내용을 요약
3. 원본 대화를 요약본으로 대체
4. 컨텍스트 공간 확보 후 대화 계속

```
[자동 압축 발생]
이전 대화가 압축되었습니다. 핵심 내용은 유지됩니다.
```

#### 자동 압축의 한계

- 세부적인 대화 내용이 손실될 수 있음
- 이전에 논의했던 코드의 정확한 줄 번호 등이 부정확해질 수 있음
- 압축 과정에서 일부 맥락이 생략될 수 있음

---

## `/compact` - 수동 압축 명령어

자동 압축을 기다리지 않고 **직접 컨텍스트를 압축**할 수 있습니다.

### 기본 사용법

```
> /compact
```

현재까지의 대화 내용을 요약하여 컨텍스트 공간을 확보합니다.

### 커스텀 지시사항과 함께 사용

```
> /compact 파일 수정 관련 내용만 유지해줘
```

특정 내용을 우선적으로 보존하도록 지시할 수 있습니다.

### `/compact` 사용이 효과적인 시점

| 상황 | 권장 |
|------|------|
| 긴 코드 분석 후 다음 작업으로 넘어갈 때 | `/compact` 사용 |
| 여러 파일을 읽은 후 실제 수정 작업 시작 전 | `/compact` 사용 |
| 대화가 20~30턴 이상 진행되었을 때 | `/compact` 사용 |
| 디버깅 세션이 길어졌을 때 | `/compact` 사용 |
| 주제를 완전히 바꿀 때 | `/clear` 사용 (압축이 아닌 초기화) |

### `/compact` vs `/clear`

| 명령어 | 동작 | 기존 맥락 |
|--------|------|-----------|
| `/compact` | 대화 내용을 요약하여 압축 | 요약 형태로 유지 |
| `/clear` | 대화 내용을 완전히 삭제 | 완전히 제거 |

## 컨텍스트를 깨끗하게 유지하는 전략

### 1. 작업 단위로 대화 나누기

하나의 세션에서 모든 작업을 하지 마세요. 작업이 바뀌면 새로운 세션을 시작하거나 `/clear`를 사용하세요.

```bash
# 세션 1: 버그 수정
claude
> src/api/users.ts의 404 에러 버그를 수정해줘
> (수정 완료 후 종료)

# 세션 2: 새 기능 개발
claude
> 사용자 프로필 이미지 업로드 기능을 추가해줘
```

### 2. 필요한 파일만 읽기

```
# 비효율적: 모든 파일을 무작정 읽기
> src 폴더의 모든 파일을 보여줘

# 효율적: 필요한 파일만 지정
> src/utils/auth.ts 파일을 보여줘
```

### 3. 큰 파일은 필요한 부분만 확인

```
# 비효율적: 10,000줄짜리 파일 전체 읽기
> app-bundle.js 파일을 보여줘

# 효율적: 특정 함수만 확인
> app-bundle.js에서 handleLogin 함수 부분만 보여줘
```

### 4. 명령어 출력이 길면 주의

```
# 비효율적: 수천 줄의 로그 전체를 컨텍스트에 넣기
> npm test 결과를 보여줘  (테스트가 100개이고 모두 출력될 때)

# 효율적: 필요한 부분만 확인
> npm test 중 실패한 테스트만 보여줘
```

### 5. 정기적으로 `/compact` 사용

긴 작업 세션에서는 중간중간 `/compact`를 사용하여 컨텍스트를 정리하세요.

```
> (여러 파일 분석 후)
> /compact 지금까지 분석 결과의 핵심만 유지해줘
> (이어서 작업)
```

## CLAUDE.md로 지속적 컨텍스트 제공하기

`CLAUDE.md`는 **매 세션마다 자동으로 로드**되는 특별한 파일입니다. 프로젝트에 대한 핵심 정보를 여기에 기록하면, 매번 같은 설명을 반복할 필요가 없습니다.

### CLAUDE.md의 위치와 우선순위

```
~/.claude/CLAUDE.md              # 전역 (모든 프로젝트에 적용)
~/project/CLAUDE.md              # 프로젝트 루트 (현재 프로젝트에 적용)
~/project/src/CLAUDE.md          # 하위 디렉터리 (해당 디렉터리 작업 시 적용)
```

### 효과적인 CLAUDE.md 작성 예시

```markdown
# 프로젝트: 온라인 서점 API

## 기술 스택
- Runtime: Node.js 20
- Framework: Express.js
- Database: PostgreSQL 15 + Prisma ORM
- Auth: JWT + bcrypt
- Test: Jest + Supertest

## 프로젝트 구조
src/
  routes/      # API 엔드포인트
  services/    # 비즈니스 로직
  models/      # Prisma 스키마 모델
  middleware/  # Express 미들웨어
  utils/       # 유틸리티 함수

## 코드 컨벤션
- 모든 함수에 TypeScript 타입 지정 필수
- 에러 핸들링은 커스텀 AppError 클래스 사용
- API 응답 형식: { success: boolean, data?: any, error?: string }
- 테스트 파일은 __tests__ 폴더에 위치

## 주요 명령어
- npm run dev: 개발 서버 (포트 3000)
- npm test: Jest 테스트
- npm run build: TypeScript 컴파일
- npx prisma migrate dev: DB 마이그레이션

## 주의사항
- .env 파일은 절대 커밋하지 말 것
- API 키는 환경 변수로만 관리
- main 브랜치에 직접 push 금지
```

### CLAUDE.md 작성 팁

1. **간결하게**: CLAUDE.md도 컨텍스트 윈도우를 사용하므로, 핵심 정보만 담기
2. **구조적으로**: 항목별로 정리하여 Claude가 쉽게 참조할 수 있게
3. **실용적으로**: 실제 작업에 필요한 정보 위주 (빌드 명령어, 컨벤션, 구조 등)
4. **최신 상태 유지**: 프로젝트가 변경되면 CLAUDE.md도 업데이트

## 비용과 컨텍스트의 관계

### 토큰 사용량과 비용

Claude Code는 API 토큰 사용량에 따라 비용이 발생합니다. 컨텍스트가 클수록 매 요청마다 더 많은 토큰이 전송됩니다.

```
입력 토큰 비용 = 컨텍스트 크기 x 요청 횟수
```

### 비용 절감 전략

| 전략 | 효과 | 설명 |
|------|------|------|
| `/compact` 정기 사용 | 높음 | 불필요한 컨텍스트 제거 |
| 작업 단위로 세션 분리 | 높음 | 각 세션의 컨텍스트를 최소화 |
| 필요한 파일만 읽기 | 중간 | 불필요한 파일 내용 방지 |
| `CLAUDE.md` 활용 | 중간 | 반복 설명 제거 |
| `/clear` 적극 사용 | 높음 | 주제 변경 시 완전 초기화 |
| `-p` 모드 활용 | 높음 | 단일 질문은 비대화형으로 |

### 비용 확인

세션이 끝나면 사용된 토큰 수가 표시됩니다. 이를 통해 작업별 비용을 추적할 수 있습니다.

## 실전 팁 모음

### 1. 집중된 작업 요청

```
# 비효율적 (컨텍스트를 많이 사용)
> 프로젝트 전체를 분석하고 모든 문제점을 알려줘

# 효율적 (컨텍스트를 적게 사용)
> src/auth 폴더의 인증 로직에서 보안 취약점을 확인해줘
```

### 2. 명확한 지시사항 제공

모호한 요청은 Claude가 더 많은 파일을 탐색하게 만들어 컨텍스트를 낭비합니다.

```
# 모호함 (Claude가 여러 파일을 탐색)
> 에러가 나는 부분을 고쳐줘

# 명확함 (필요한 파일만 접근)
> src/routes/users.ts:42에서 TypeError가 발생해.
> userId가 string인데 number로 비교하고 있어서 그래.
```

### 3. 큰 작업을 작은 단위로 분리

```bash
# 하나의 긴 세션 대신
# 여러 짧은 세션으로 분리

# 세션 1: 분석
claude -p "src/api 폴더의 구조와 각 파일의 역할을 요약해줘" > analysis.txt

# 세션 2: 특정 파일 수정
claude
> src/api/users.ts를 리팩토링해줘 (analysis.txt 참고)

# 세션 3: 테스트 추가
claude
> src/api/users.ts에 대한 유닛 테스트를 작성해줘
```

### 4. 컨텍스트 사용량 체감 기준

| 작업 | 대략적인 토큰 사용량 |
|------|---------------------|
| 짧은 대화 (5턴) | 5,000 ~ 10,000 |
| 파일 1개 읽기 (100줄) | 500 ~ 1,000 |
| 파일 1개 읽기 (1,000줄) | 5,000 ~ 10,000 |
| npm test 결과 | 1,000 ~ 10,000 |
| 깊은 분석 대화 (30턴) | 50,000 ~ 100,000 |

### 5. 세션 재개 활용

작업을 중간에 멈추고 나중에 이어해야 할 때:

```bash
# 세션 종료 후 나중에
claude --continue  # 가장 최근 세션 이어하기
```

단, 재개된 세션에서도 컨텍스트가 누적되므로, 이전 맥락이 충분히 남아있는지 확인하세요. 필요하면 `/compact`로 정리 후 진행합니다.
