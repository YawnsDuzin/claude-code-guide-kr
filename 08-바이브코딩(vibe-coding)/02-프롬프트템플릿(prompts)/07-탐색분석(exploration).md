# 07. 탐색/분석 프롬프트

> 낯선 코드베이스를 처음 만졌을 때. "고치기 전에 읽기"를 에이전트에게 강제.

## 두 가지 시나리오

1. **새 프로젝트에 합류** — 전체 구조 파악
2. **낯선 영역 수정 전** — 해당 영역만 빠르게 파악

---

## 템플릿 A: 코드베이스 전체 오리엔테이션

```markdown
## 역할
너는 이 프로젝트에 **방금 합류한 시니어 개발자**다.
아직 코드를 수정하지 마라. **읽기만** 한다.

## 참조
- 프로젝트 루트
- 있으면: CLAUDE.md, README.md, docs/

## 지시
다음 7가지를 순서대로 조사해서 답하라.

1. **프로젝트 목적**: README/PRD 기반 1~2문장
2. **주요 기술 스택**: 언어, 프레임워크, DB, 인프라
3. **디렉토리 구조 요약**: 5~10줄 트리 + 각 폴더 1줄 역할
4. **진입점 (Entry Points)**: 서버/웹/CLI 각각의 시작 파일
5. **핵심 도메인 개념**: 코드에 반복 등장하는 도메인 용어 5~10개
6. **빌드/테스트/실행 방법**: 명령어만
7. **위험 지점 / 불확실성**: 문서가 없거나 일관성 없는 영역

## 출력 형식
- 각 항목 짧게 (총 응답 800단어 이하)
- 추측 금지. 근거 파일 경로 병기.
- 마지막에 "이해 못 한 것 3개" 질문 리스트

⚠️ 파일을 **수정하지 마라**. 이 작업은 읽기 전용이다.
```

---

## 템플릿 B: 특정 영역 집중 분석

건드려야 할 영역이 정해진 경우.

```markdown
## 역할
너는 이 프로젝트의 시니어 개발자다.
특정 모듈을 수정하려 하는데, 먼저 맥락을 파악한다. **읽기만** 한다.

## 참조
- CLAUDE.md
- 대상 영역: src/<경로>/
- 관련 문서: docs/<...>

## 지시
다음 6가지를 조사하라.

1. **이 영역의 역할**: 한 문단 이내
2. **공개 API**: 이 모듈을 쓰는 외부 호출자 목록 (파일:심볼)
3. **의존성**: 이 모듈이 의존하는 다른 모듈 / 외부 라이브러리
4. **데이터 흐름**: 입력 → 변환 → 출력을 3~5단계로
5. **기존 테스트**: 어떤 시나리오가 덮이고 어떤 게 안 덮이는가
6. **수정 시 주의점**: 공개 시그니처, 전역 상태, 부수 효과

## 출력 형식
- 각 항목 5줄 이하
- 파일 경로 필수
- Mermaid로 데이터 흐름 그리기 (선택)
- 마지막에 "수정 시 건드릴 파일 예상 리스트"

⚠️ 수정하지 마라.
```

---

## "읽기 전용" 강제의 중요성

탐색 단계에서 에이전트가 **"겸사겸사"** 코드를 고치기 시작하면 상태가 오염됩니다. 프롬프트에 **"수정 금지"** 를 굵게 넣으세요. 에이전트는 지시를 따릅니다.

---

## 응용: 이 결과를 CLAUDE.md로

템플릿 A의 결과가 좋으면 그대로 `CLAUDE.md` 초안으로 씁니다. 다음 프롬프트가 도움이 됩니다.

```
위에서 네가 파악한 내용을 CLAUDE.md 형식으로 정리해줘.
섹션은 다음 7개:
1. 프로젝트 한 줄 요약
2. 기술 스택
3. 디렉토리 구조
4. 빌드/테스트/실행 명령
5. 코딩 규칙 (네가 코드에서 관찰한 패턴 기반)
6. 금지 사항 (네가 관찰한 안티패턴 역)
7. 참조 문서 경로

추측 금지. 근거 없는 항목은 "(추가 필요)"로 남긴다.
```
